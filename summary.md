# 심천지사 영수증 정산 자동화 웹앱 (SROT) 작업 요약

## 1. 프로젝트 목표
- **목적:** 중국 심천지사에서 발생하는 증치세 영수증(VAT Invoice) 및 기타 증빙 서류의 데이터를 자동 추출하여 엑셀 정산 양식에 맞게 변환하는 사내 자동화 웹앱 개발.
- **주요 기능:** PDF/이미지 업로드 -> OCR 텍스트 추출 -> 핵심 데이터(유형, 금액, 날짜, 판매자 등) 파싱 -> 웹 UI 검토 및 수정 -> 정산용 엑셀 다운로드.

## 2. 발생했던 문제점 (Tesseract OCR의 한계)
- **인식률 저하:** 중국 영수증 특유의 복잡한 배경(D8 GlobalChip 워터마크 등), 희미한 도장, 표 양식으로 인해 기존 규칙 기반의 `Tesseract OCR` 엔진이 글자를 심하게 뭉개거나 노이즈로 인식함.
- **자동 회전(OSD) 실패:** 스캔 시 90도, 180도로 누워있는 이미지를 바로잡는 과정에서, 해상도 다운스케일링과 OpenCV 전처리(이진화 등)가 결합되며 한자 데이터가 완전히 소실되는 현상 발생.
- **추출 누락:** 금액과 영수증 유형 일부만 간헐적으로 추출되고, 가장 중요한 '날짜'와 '판매자 명'의 정규식 매칭이 불가능한 상태 지속.

## 3. 해결 방안: AI 기반 OCR 도입 (RapidOCR)
사용자의 요청에 따라 Tesseract를 폐기하고, 상용 수준의 압도적인 중국어 인식률을 자랑하는 AI 기반 OCR로 백엔드 파이프라인을 전면 교체함.

### 💡 의사결정 내용 (PaddleOCR vs RapidOCR)
- **제약 사항:** 유료 클라우드 API(LLM 등) 사용 불가, 오프라인 로컬 환경에서 무료로 동작해야 함.
- **초기 검토 (PaddleOCR):** 바이두의 강력한 모델이지만, 로컬 PC에 1~2GB에 달하는 `paddlepaddle` 딥러닝 프레임워크를 설치해야 하여 환경이 심하게 무거워지는 단점 발생.
- **최종 도입 (RapidOCR):** 
  - PaddleOCR의 최신 훈련 모델(PP-OCRv4)의 "가중치(두뇌)"는 100% 동일하게 가져오되, 실행 엔진만 마이크로소프트의 가벼운 `ONNX Runtime`으로 교체한 **RapidOCR** 채택.
  - 이를 통해 무거운 딥러닝 패키지 설치 없이 CPU만으로 가볍고 빠르게 동작하면서도, PaddleOCR과 완전히 동일한 질적 정확도를 달성.

## 4. 백엔드 적용 및 고도화 내역
1. **의존성 업데이트:** `requirements.txt`에서 `pytesseract`를 제거하고 `rapidocr_onnxruntime` 설치 완료.
2. **코드 전면 개편 (`ocr_engine.py`):** 
   - 기존의 불필요하고 불안정했던 OpenCV 전처리 로직(Deskew, CLAHE, Otsu 이진화, OSD 회전 등)을 전부 삭제.
   - RapidOCR 엔진 객체 하나로 텍스트 위치 감지, 회전 보정, 한자 인식을 한 번에 처리하도록 단순화 및 안정화.
3. **파서 정규식 최적화 (`receipt_parser.py`):** 
   - 텍스트 줄바꿈 구조가 AI OCR에 맞게 정돈됨에 따라, 판매자(`seller`) 및 날짜(`date`)를 완벽하게 잡아내도록 정규식 패턴 미세 조정.
4. **캐시 무효화 완비:** 이전 실패 데이터가 잔류하여 UI에 오류를 일으키는 현상을 방지하기 위해 `ocr_cache.json` 초기화 적용.

## 5. 배치 프로세스 및 매핑 시스템 구축 (최신 업데이트)
1. **폴더 기반 일괄 정산 (Batch Processing):**
   - 기존 웹앱 업로드 방식 대신 `영수증 보관소` 내 폴더(YYYY-MM) 기반의 일괄 처리 스크립트(`batch_processor.py`) 구축 완료.
   - 영수증 이미지들의 순차적 리네이밍(증빙번호 매칭) 및 다중 페이지 PDF 분할 기능(fitz 활용) 탑재.
2. **자동 매핑 엔진 (`receipt_mapper.py`):**
   - 과거 엑셀 정산 데이터(양식 파일의 숨겨진 시트)를 텍스트 딕셔너리로 추출하여 오프라인 학습.
   - 텍스트 유사도 알고리즘(`difflib`)을 통해 OCR로 읽은 텍스트를 대계정/소계정/표준내역으로 자동 매핑.
3. **엑셀 출력 고도화 (`excel_exporter.py`):**
   - 매핑에 실패한 알 수 없는 항목은 **빨간색 배경(PatternFill)**으로 시각적 강조 처리.
   - 담당자 이름 추출 시 직급(책임, 수석 등)을 정규식으로 자동 제거하여 깔끔한 포맷팅 유지.

## 6. 현재 결과 및 상태
- 가로로 90도 눕혀진 상태에 워터마크가 찍힌 악조건의 영수증에서도 유형, 금액, 날짜, 판매자 데이터를 단 한 글자의 오타 없이 추출 성공.
- 과거 5월 스캔본(다중 페이지 PDF 11개, 총 약 220장)에 대해 자동 이미지 분할 및 백그라운드 매핑 테스트 진행 성공.

## 7. 월별 정산 대시보드 고도화 완료 (Phase 2)
- **과거 데이터 학습 및 매핑 규칙 생성 (`learn_past_data.py`)**: 80개 이상의 과거 정산 시트에서 데이터를 추출하여 대계정/소계정 자동 매핑 사전을 구축 완료. (`account_mapping.json`)
- **백엔드 아키텍처 개편 (`app.py`)**: 기존 단일 JSON 의존성에서 벗어나 `영수증 보관소/{YYYY-MM}` 형태의 월별 디렉토리 구조 및 REST API(GET, PUT, DELETE) 구축 완료.
- **실시간 엑셀 동기화 (`excel_exporter.py`)**: 영수증 내용이 수정되거나 삭제될 때마다 해당 월의 엑셀 정산서 파일을 즉시 재생성하여 덮어쓰도록(Row 21 이후 초기화 로직 추가) 완벽 구현.
- **영수증 삭제 및 순차 정렬**: 대시보드에서 영수증을 삭제할 경우 이후 영수증들의 일련번호(`evidence_no`)와 실제 이미지 파일명(`001.png`, `002.png`...)을 순차적으로 다시 정렬(Renaming)하는 백엔드 로직 반영.
- **프론트엔드 UI/UX 업그레이드 (`index.html`, `style.css`)**:
  - 사용자 취향에 맞춘 **글래스모피즘(Glassmorphism)** 테마와 파스텔 네온 색상을 적용하여 매우 세련된 프리미엄 디자인 완성.
  - 헤더 영역에 **📅 정산 월 선택** 드롭다운을 추가하여 손쉽게 월별 데이터셋 전환이 가능하도록 구성.
  - 대계정/소계정 자동 매핑 연동 처리를 완료했으며, 매핑되지 않은(새로운) 내역에 대해서는 붉은색 테두리와 뱃지(`⚠️ 계정 확인 필요`)로 명확히 시각적 경고가 제공되도록 스타일 보강.
- **프론트엔드 API 연동 (`app.js`)**: 월별 데이터 조회, 이미지 로딩 경로 최적화, 영수증 편집(저장)/삭제 버튼을 구축된 REST API와 완벽하게 연결.

## 8. 5월 스캔본 대규모 배치 처리 및 백그라운드 분석 (현황 보고)
### 1) 어제 분석 진행 상황 (중단 원인 및 캐시 보존)
- 일괄 처리 스크립트(`batch_processor.py`)가 다중 페이지 PDF들을 **총 220장의 이미지(`001.png` ~ `220.png`)로 완벽히 분할**하고 순차 리네이밍을 성공적으로 완료했습니다.
- 이후 AI 기반 OCR(RapidOCR) 엔진이 220장에 대한 텍스트 추출을 백그라운드에서 진행하던 중, 방대한 분석 양으로 인해 대화 세션이 종료되며 스크립트가 도중에 중단되었습니다. (최종 `정산내역_2026-05.xlsx` 및 `data.json` 미생성 상태 발생)

### 2) 작업 데이터 100% 생존 (OCR 캐싱 시스템의 위력)
- `ocr_engine.py`에 탑재된 이미지 해시(SHA-256) 기반의 **실시간 캐싱 시스템(`data/ocr_cache.json`)** 덕분에 어제 중단되기 직전까지 AI가 분석했던 모든 텍스트 추출 결과가 유실 없이 안전하게 보존되었습니다.

### 3) 현재 진행 조치 사항 (작업 재개 및 대시보드 연동 대기)
- 백그라운드에서 `2026-05` 폴더에 대한 일괄 처리 스크립트를 다시 가동했습니다. 어제 이미 분석이 끝난 이미지는 캐시에서 0.1초 만에 즉시 불러오고, 남은 이미지만 추가로 고속 분석 중입니다.
- 분석이 완료되면 어제 학습한 80개의 매핑 딕셔너리(`account_mapping.json`)를 참조해 **모든 영수증이 대/소계정으로 자동 매핑**된 상태로 엑셀 및 JSON 파일이 생성됩니다.
- 생성 직후 사용자는 **대시보드(http://127.0.0.1:8080)**의 "정산 월 선택"에서 `2026-05`를 section을 선택하여, 220장의 방대한 영수증 데이터를 웹 UI 상에서 편하게 감상 및 편집하고 즉시 엑셀과 동기화할 수 있습니다.

## 10. OCR 인식률 극대화 및 급여 명세서 자동 분할 (최신 업데이트)
### 1) OCR 전처리(Preprocessing) 강화
- 기존 `rapidocr_onnxruntime` 엔진의 장점(빠른 속도, 가벼운 의존성)을 그대로 유지하면서 인식률을 극한으로 끌어올리기 위해 **OpenCV 이미지 전처리 파이프라인**을 `ocr_engine.py`에 새롭게 도입했습니다.
- **구현된 파이프라인**: `Grayscale 변환` ➔ `적응형 히스토그램 평활화 (CLAHE)` ➔ `가우시안 블러 노이즈 제거` ➔ `해상도 최적화(업스케일링/다운스케일링)` ➔ `샤프닝(윤곽선 강화)`
- 전자 영수증 등 원본 디지털 파일이 들어올 경우 원본의 선명함을 손상시키지 않고, 흐릿하거나 빛 번짐이 있는 스캔 영수증만 스마트하게 화질을 개선하여 문자를 뚜렷하게 복원합니다.

### 2) 은행 이체증명서(급여/수수료) 자동 행 분할
- 중국 은행 영수증(银行回单) 내에 **급여(工资)** 키워드와 하단의 **수수료(手续费)** 금액이 함께 기재된 경우, 시스템이 이를 인지하여 엑셀 정산 시 **자동으로 2줄(급여 1줄, 수수료 1줄)**로 분할하여 기입하도록 `receipt_parser.py` 및 `batch_processor.py` 로직을 업그레이드했습니다.
- 이로써 기존 5월 증빙(61-62번) 뿐만 아니라, 향후 6월 이후 영수증에서도 급여 송금 내역이 업로드되면 자동으로 2줄 분리 입력되어 수기 보정의 번거로움이 사라졌습니다.


## 11. 이체 총액 검증 및 출금일자 일괄 적용 시스템 탑재 (최신 업데이트)
### 1) 출금일자(B열)와 증빙일자(C열) 역할 분리
- 기존에는 두 열에 동일한 OCR 날짜가 기입되었으나, **C열(증빙일자)**에는 개별 영수증 날짜를, **B열(출금일자)**에는 소속 담당자의 은행 이체증명서 날짜를 찾아 일괄 덮어쓰도록 로직을 분리했습니다.
- 이체증명서가 없는 권유석 직원의 경우에만 B열과 C열을 동일하게 채우는 예외 처리를 반영했습니다.

### 2) 출금 총계 자동 검증(Validation)
- 은행 이체증명서가 인식되면 원금(출금총계)은 엑셀에 지출 내역으로 추가하지 않고 오직 **은행수수료 1줄**만 기입하도록 최적화했습니다.
- 대신, `이체증명서의 원금`과 `해당 직원의 개별 영수증 합계(수수료 제외)`를 백그라운드에서 실시간 비교하여, 금액이 1위안이라도 다를 경우 출력창에 **[⚠️ 경고]** 메시지를 띄워줍니다. 이를 통해 영수증 누락이나 OCR 인식 오류를 즉시 잡아낼 수 있습니다.

## 12. 세부 정산 데이터 매핑 및 포맷팅 개선 (최신 업데이트)
### 1) 알 수 없는 중국어 가맹점 자동 한글 변환
- `receipt_mapper.py`에 중국의 택시 운수회사(`出租`, `滴滴`) 및 고속도로/법인명(`高速`, `公司`, `股份` 등) 키워드를 대거 추가하여, OCR이 증치세 발포로 인식한 중국어 가맹점명이 내역에 그대로 노출되는 문제를 해결했습니다.
- 이제 톨비 및 택시비 관련 중국어 가맹점명이 포함된 영수증은 대계정(여비교통비), 소계정(교통비), 내역(교통비/톨비)으로 완벽하게 **자동 한글 매핑**됩니다.

### 2) 출금일자(B열) 엑셀 표기 서식 깔끔화
- 기존 V4 엑셀 리포트에서 출금일자가 시간(`00:00:00`)까지 포함된 포맷으로 출력되어 가독성이 떨어지던 현상을 해소했습니다.
- `excel_exporter.py`를 수정하여 모든 출금일자와 증빙일자(C열)가 엑셀 상에 깔끔한 문자열(`YYYY-MM-DD` 형식)로 일관되게 삽입되도록 정돈하였습니다.
- **참고**: 은행 이체증명서가 존재하지 않는 권유석 님은 예외 규칙에 따라 각자의 증빙일자가 출금일자로 대체 적용되었습니다.

## 13. 파표 세무코드 검증 및 텍스트 파싱 보완 (최신 업데이트)
### 1) 파표(증치세 발포) 세무코드 자동 검증 기능 탑재
- `receipt_parser.py` 내에 파표 텍스트 추출 시, 회사의 고유 세무코드(`91440300MAELRTJ5XE`)가 존재하는지 스캔하는 로직을 추가했습니다.
- 추출 결과를 기반으로 `excel_exporter.py`를 통해 엑셀의 S열(지출증빙 비고)에 **"파표 회사코드 일치"** 또는 **"파표 회사코드 불일치/누락"**(빨간 배경색)으로 즉각적인 시각적 피드백이 출력되도록 개선했습니다.

### 2) 숫자 콤마(,) 오인식 파싱 오류 완벽 해결
- OCR 엔진이 금액의 천 단위 구분 기호인 콤마를 온점(.)으로 잘못 인식하여 `12,405.76`을 `12.405.76`처럼 출력하는 경우, 기존 파서에서는 `ValueError`가 발생하여 금액을 누락하는 치명적인 버그가 있었습니다.
- 소수점이 2개 이상일 때 맨 마지막 소수점만 보존하고 앞의 소수점들은 제거하는 스마트 클리닝 알고리즘을 `clean_amount()`에 도입하여, 어떠한 오인식 상태에서도 금액을 정확한 float(실수) 값으로 정제하도록 패치했습니다.

### 3) 학습 모델 오버피팅(Overfitting) 해결 및 분류 안정화
- 5월의 과거 데이터(급여 송금 이체증)를 기반으로 학습한 AI 모델이, 모든 일반 신한은행 이체증명서를 '급여'로 강제 오분류해버리는 오버피팅 현상을 발견했습니다.
- `batch_processor.py`에서 은행 이체증명서(`银行回单`) 유형으로 판독된 증빙은 억지로 과거 학습 데이터를 유추하지 않고, 파서가 분석한 '은행 거래' 내역을 그대로 보존하도록 예외 처리했습니다. (단, 영수증 텍스트 내에 실제로 '급여(工资)'가 적혀 있는 경우에만 급여로 인식함)
# 심천지사 영수증 정산 자동화 웹앱 (SROT) 작업 요약

## 1. 프로젝트 목표
- **목적:** 중국 심천지사에서 발생하는 증치세 영수증(VAT Invoice) 및 기타 증빙 서류의 데이터를 자동 추출하여 엑셀 정산 양식에 맞게 변환하는 사내 자동화 웹앱 개발.
- **주요 기능:** PDF/이미지 업로드 -> OCR 텍스트 추출 -> 핵심 데이터(유형, 금액, 날짜, 판매자 등) 파싱 -> 웹 UI 검토 및 수정 -> 정산용 엑셀 다운로드.

## 2. 발생했던 문제점 (Tesseract OCR의 한계)
- **인식률 저하:** 중국 영수증 특유의 복잡한 배경(D8 GlobalChip 워터마크 등), 희미한 도장, 표 양식으로 인해 기존 규칙 기반의 `Tesseract OCR` 엔진이 글자를 심하게 뭉개거나 노이즈로 인식함.
- **자동 회전(OSD) 실패:** 스캔 시 90도, 180도로 누워있는 이미지를 바로잡는 과정에서, 해상도 다운스케일링과 OpenCV 전처리(이진화 등)가 결합되며 한자 데이터가 완전히 소실되는 현상 발생.
- **추출 누락:** 금액과 영수증 유형 일부만 간헐적으로 추출되고, 가장 중요한 '날짜'와 '판매자 명'의 정규식 매칭이 불가능한 상태 지속.

## 3. 해결 방안: AI 기반 OCR 도입 (RapidOCR)
사용자의 요청에 따라 Tesseract를 폐기하고, 상용 수준의 압도적인 중국어 인식률을 자랑하는 AI 기반 OCR로 백엔드 파이프라인을 전면 교체함.

### 💡 의사결정 내용 (PaddleOCR vs RapidOCR)
- **제약 사항:** 유료 클라우드 API(LLM 등) 사용 불가, 오프라인 로컬 환경에서 무료로 동작해야 함.
- **초기 검토 (PaddleOCR):** 바이두의 강력한 모델이지만, 로컬 PC에 1~2GB에 달하는 `paddlepaddle` 딥러닝 프레임워크를 설치해야 하여 환경이 심하게 무거워지는 단점 발생.
- **최종 도입 (RapidOCR):** 
  - PaddleOCR의 최신 훈련 모델(PP-OCRv4)의 "가중치(두뇌)"는 100% 동일하게 가져오되, 실행 엔진만 마이크로소프트의 가벼운 `ONNX Runtime`으로 교체한 **RapidOCR** 채택.
  - 이를 통해 무거운 딥러닝 패키지 설치 없이 CPU만으로 가볍고 빠르게 동작하면서도, PaddleOCR과 완전히 동일한 질적 정확도를 달성.

## 4. 백엔드 적용 및 고도화 내역
1. **의존성 업데이트:** `requirements.txt`에서 `pytesseract`를 제거하고 `rapidocr_onnxruntime` 설치 완료.
2. **코드 전면 개편 (`ocr_engine.py`):** 
   - 기존의 불필요하고 불안정했던 OpenCV 전처리 로직(Deskew, CLAHE, Otsu 이진화, OSD 회전 등)을 전부 삭제.
   - RapidOCR 엔진 객체 하나로 텍스트 위치 감지, 회전 보정, 한자 인식을 한 번에 처리하도록 단순화 및 안정화.
3. **파서 정규식 최적화 (`receipt_parser.py`):** 
   - 텍스트 줄바꿈 구조가 AI OCR에 맞게 정돈됨에 따라, 판매자(`seller`) 및 날짜(`date`)를 완벽하게 잡아내도록 정규식 패턴 미세 조정.
4. **캐시 무효화 완비:** 이전 실패 데이터가 잔류하여 UI에 오류를 일으키는 현상을 방지하기 위해 `ocr_cache.json` 초기화 적용.

## 5. 배치 프로세스 및 매핑 시스템 구축 (최신 업데이트)
1. **폴더 기반 일괄 정산 (Batch Processing):**
   - 기존 웹앱 업로드 방식 대신 `영수증 보관소` 내 폴더(YYYY-MM) 기반의 일괄 처리 스크립트(`batch_processor.py`) 구축 완료.
   - 영수증 이미지들의 순차적 리네이밍(증빙번호 매칭) 및 다중 페이지 PDF 분할 기능(fitz 활용) 탑재.
2. **자동 매핑 엔진 (`receipt_mapper.py`):**
   - 과거 엑셀 정산 데이터(양식 파일의 숨겨진 시트)를 텍스트 딕셔너리로 추출하여 오프라인 학습.
   - 텍스트 유사도 알고리즘(`difflib`)을 통해 OCR로 읽은 텍스트를 대계정/소계정/표준내역으로 자동 매핑.
3. **엑셀 출력 고도화 (`excel_exporter.py`):**
   - 매핑에 실패한 알 수 없는 항목은 **빨간색 배경(PatternFill)**으로 시각적 강조 처리.
   - 담당자 이름 추출 시 직급(책임, 수석 등)을 정규식으로 자동 제거하여 깔끔한 포맷팅 유지.

## 6. 현재 결과 및 상태
- 가로로 90도 눕혀진 상태에 워터마크가 찍힌 악조건의 영수증에서도 유형, 금액, 날짜, 판매자 데이터를 단 한 글자의 오타 없이 추출 성공.
- 과거 5월 스캔본(다중 페이지 PDF 11개, 총 약 220장)에 대해 자동 이미지 분할 및 백그라운드 매핑 테스트 진행 성공.

## 7. 월별 정산 대시보드 고도화 완료 (Phase 2)
- **과거 데이터 학습 및 매핑 규칙 생성 (`learn_past_data.py`)**: 80개 이상의 과거 정산 시트에서 데이터를 추출하여 대계정/소계정 자동 매핑 사전을 구축 완료. (`account_mapping.json`)
- **백엔드 아키텍처 개편 (`app.py`)**: 기존 단일 JSON 의존성에서 벗어나 `영수증 보관소/{YYYY-MM}` 형태의 월별 디렉토리 구조 및 REST API(GET, PUT, DELETE) 구축 완료.
- **실시간 엑셀 동기화 (`excel_exporter.py`)**: 영수증 내용이 수정되거나 삭제될 때마다 해당 월의 엑셀 정산서 파일을 즉시 재생성하여 덮어쓰도록(Row 21 이후 초기화 로직 추가) 완벽 구현.
- **영수증 삭제 및 순차 정렬**: 대시보드에서 영수증을 삭제할 경우 이후 영수증들의 일련번호(`evidence_no`)와 실제 이미지 파일명(`001.png`, `002.png`...)을 순차적으로 다시 정렬(Renaming)하는 백엔드 로직 반영.
- **프론트엔드 UI/UX 업그레이드 (`index.html`, `style.css`)**:
  - 사용자 취향에 맞춘 **글래스모피즘(Glassmorphism)** 테마와 파스텔 네온 색상을 적용하여 매우 세련된 프리미엄 디자인 완성.
  - 헤더 영역에 **📅 정산 월 선택** 드롭다운을 추가하여 손쉽게 월별 데이터셋 전환이 가능하도록 구성.
  - 대계정/소계정 자동 매핑 연동 처리를 완료했으며, 매핑되지 않은(새로운) 내역에 대해서는 붉은색 테두리와 뱃지(`⚠️ 계정 확인 필요`)로 명확히 시각적 경고가 제공되도록 스타일 보강.
- **프론트엔드 API 연동 (`app.js`)**: 월별 데이터 조회, 이미지 로딩 경로 최적화, 영수증 편집(저장)/삭제 버튼을 구축된 REST API와 완벽하게 연결.

## 8. 5월 스캔본 대규모 배치 처리 및 백그라운드 분석 (현황 보고)
### 1) 어제 분석 진행 상황 (중단 원인 및 캐시 보존)
- 일괄 처리 스크립트(`batch_processor.py`)가 다중 페이지 PDF들을 **총 220장의 이미지(`001.png` ~ `220.png`)로 완벽히 분할**하고 순차 리네이밍을 성공적으로 완료했습니다.
- 이후 AI 기반 OCR(RapidOCR) 엔진이 220장에 대한 텍스트 추출을 백그라운드에서 진행하던 중, 방대한 분석 양으로 인해 대화 세션이 종료되며 스크립트가 도중에 중단되었습니다. (최종 `정산내역_2026-05.xlsx` 및 `data.json` 미생성 상태 발생)

### 2) 작업 데이터 100% 생존 (OCR 캐싱 시스템의 위력)
- `ocr_engine.py`에 탑재된 이미지 해시(SHA-256) 기반의 **실시간 캐싱 시스템(`data/ocr_cache.json`)** 덕분에 어제 중단되기 직전까지 AI가 분석했던 모든 텍스트 추출 결과가 유실 없이 안전하게 보존되었습니다.

### 3) 현재 진행 조치 사항 (작업 재개 및 대시보드 연동 대기)

## 10. OCR 인식률 극대화 및 급여 명세서 자동 분할 (최신 업데이트)
### 1) OCR 전처리(Preprocessing) 강화
- 기존 `rapidocr_onnxruntime` 엔진의 장점(빠른 속도, 가벼운 의존성)을 그대로 유지하면서 인식률을 극한으로 끌어올리기 위해 **OpenCV 이미지 전처리 파이프라인**을 `ocr_engine.py`에 새롭게 도입했습니다.
- **구현된 파이프라인**: `Grayscale 변환` ➔ `적응형 히스토그램 평활화 (CLAHE)` ➔ `가우시안 블러 노이즈 제거` ➔ `해상도 최적화(업스케일링/다운스케일링)` ➔ `샤프닝(윤곽선 강화)`
- 전자 영수증 등 원본 디지털 파일이 들어올 경우 원본의 선명함을 손상시키지 않고, 흐릿하거나 빛 번짐이 있는 스캔 영수증만 스마트하게 화질을 개선하여 문자를 뚜렷하게 복원합니다.

### 2) 은행 이체증명서(급여/수수료) 자동 행 분할
- 중국 은행 영수증(银行回单) 내에 **급여(工资)** 키워드와 하단의 **수수료(手续费)** 금액이 함께 기재된 경우, 시스템이 이를 인지하여 엑셀 정산 시 **자동으로 2줄(급여 1줄, 수수료 1줄)**로 분할하여 기입하도록 `receipt_parser.py` 및 `batch_processor.py` 로직을 업그레이드했습니다.
- 이로써 기존 5월 증빙(61-62번) 뿐만 아니라, 향후 6월 이후 영수증에서도 급여 송금 내역이 업로드되면 자동으로 2줄 분리 입력되어 수기 보정의 번거로움이 사라졌습니다.


## 11. 이체 총액 검증 및 출금일자 일괄 적용 시스템 탑재 (최신 업데이트)
### 1) 출금일자(B열)와 증빙일자(C열) 역할 분리
- 기존에는 두 열에 동일한 OCR 날짜가 기입되었으나, **C열(증빙일자)**에는 개별 영수증 날짜를, **B열(출금일자)**에는 소속 담당자의 은행 이체증명서 날짜를 찾아 일괄 덮어쓰도록 로직을 분리했습니다.
- 이체증명서가 없는 권유석 직원의 경우에만 B열과 C열을 동일하게 채우는 예외 처리를 반영했습니다.

### 2) 출금 총계 자동 검증(Validation)
- 은행 이체증명서가 인식되면 원금(출금총계)은 엑셀에 지출 내역으로 추가하지 않고 오직 **은행수수료 1줄**만 기입하도록 최적화했습니다.
- 대신, `이체증명서의 원금`과 `해당 직원의 개별 영수증 합계(수수료 제외)`를 백그라운드에서 실시간 비교하여, 금액이 1위안이라도 다를 경우 출력창에 **[⚠️ 경고]** 메시지를 띄워줍니다. 이를 통해 영수증 누락이나 OCR 인식 오류를 즉시 잡아낼 수 있습니다.

## 12. 세부 정산 데이터 매핑 및 포맷팅 개선 (최신 업데이트)
### 1) 알 수 없는 중국어 가맹점 자동 한글 변환
- `receipt_mapper.py`에 중국의 택시 운수회사(`出租`, `滴滴`) 및 고속도로/법인명(`高速`, `公司`, `股份` 등) 키워드를 대거 추가하여, OCR이 증치세 발포로 인식한 중국어 가맹점명이 내역에 그대로 노출되는 문제를 해결했습니다.
- 이제 톨비 및 택시비 관련 중국어 가맹점명이 포함된 영수증은 대계정(여비교통비), 소계정(교통비), 내역(교통비/톨비)으로 완벽하게 **자동 한글 매핑**됩니다.

### 2) 출금일자(B열) 엑셀 표기 서식 깔끔화
- 기존 V4 엑셀 리포트에서 출금일자가 시간(`00:00:00`)까지 포함된 포맷으로 출력되어 가독성이 떨어지던 현상을 해소했습니다.
- `excel_exporter.py`를 수정하여 모든 출금일자와 증빙일자(C열)가 엑셀 상에 깔끔한 문자열(`YYYY-MM-DD` 형식)로 일관되게 삽입되도록 정돈하였습니다.
- **참고**: 은행 이체증명서가 존재하지 않는 권유석 님은 예외 규칙에 따라 각자의 증빙일자가 출금일자로 대체 적용되었습니다.

## 13. 파표 세무코드 검증 및 텍스트 파싱 보완 (최신 업데이트)
### 1) 파표(증치세 발포) 세무코드 자동 검증 기능 탑재
- `receipt_parser.py` 내에 파표 텍스트 추출 시, 회사의 고유 세무코드(`91440300MAELRTJ5XE`)가 존재하는지 스캔하는 로직을 추가했습니다.
- 추출 결과를 기반으로 `excel_exporter.py`를 통해 엑셀의 S열(지출증빙 비고)에 **"파표 회사코드 일치"** 또는 **"파표 회사코드 불일치/누락"**(빨간 배경색)으로 즉각적인 시각적 피드백이 출력되도록 개선했습니다.

### 2) 숫자 콤마(,) 오인식 파싱 오류 완벽 해결
- OCR 엔진이 금액의 천 단위 구분 기호인 콤마를 온점(.)으로 잘못 인식하여 `12,405.76`을 `12.405.76`처럼 출력하는 경우, 기존 파서에서는 `ValueError`가 발생하여 금액을 누락하는 치명적인 버그가 있었습니다.
- 소수점이 2개 이상일 때 맨 마지막 소수점만 보존하고 앞의 소수점들은 제거하는 스마트 클리닝 알고리즘을 `clean_amount()`에 도입하여, 어떠한 오인식 상태에서도 금액을 정확한 float(실수) 값으로 정제하도록 패치했습니다.

### 3) 학습 모델 오버피팅(Overfitting) 해결 및 분류 안정화
- 5월의 과거 데이터(급여 송금 이체증)를 기반으로 학습한 AI 모델이, 모든 일반 신한은행 이체증명서를 '급여'로 강제 오분류해버리는 오버피팅 현상을 발견했습니다.
- `batch_processor.py`에서 은행 이체증명서(`银行回单`) 유형으로 판독된 증빙은 억지로 과거 학습 데이터를 유추하지 않고, 파서가 분석한 '은행 거래' 내역을 그대로 보존하도록 예외 처리했습니다. (단, 영수증 텍스트 내에 실제로 '급여(工资)'가 적혀 있는 경우에만 급여로 인식함)

## 15. OCR 예외 패턴 대응 및 웹앱 확장 (최신 업데이트)

### 3) 학습 모델 오버라이딩(Overriding) 방지
- 과거 5월 데이터(머신러닝 학습 모델)가 '택시비'를 '출장비/해외'로 유추하여 일관성을 해치는 현상을 방지하기 위해, **수동 지정 키워드(MANUAL_KEYWORDS)의 우선순위를 가장 높게(Step 0) 격상**했습니다.

### 4) 웹앱 및 파서 영수증 유형(Type) 4종 추가
- 프론트엔드(웹앱 드롭다운) 및 백엔드(파이썬 파서) 양쪽에 사용자의 요청을 반영하여 다음 **4가지 신규 영수증 유형**을 공식 추가했습니다:

## 16. 웹앱 V2 전면 리디자인 (2026-07-03)

### 배경
엑셀 자동화 파이프라인이 안정화됨에 따라, 실제 운영 흐름이 **"파이썬 배치 실행 → 웹앱 검토/수정 → 엑셀 저장"** 으로 정착되었습니다.  
이에 따라 불필요한 기능(업로드/내보내기 탭)을 제거하고, **검토 및 정산 확정 중심의 고급 관리 대시보드**로 전면 리디자인했습니다.

### 사전 PRD 작성 → 사용자 승인 → 구현 순서로 진행
`case_study_webapp_v2_prd.md` 파일에 사례 공유용 전체 PRD를 기록.

---

### 제거된 기능

| 제거 항목 | 이유 |
|----------|------|
| 영수증 업로드 탭 | 배치 처리로 완전 대체 완료 |
| SROT 로고 + 구 시스템명 | 브랜드 교체 |
| OCR 원본 텍스트 필드 | 사용자 불필요 (개발용 디버그) |

### 추가/변경된 기능

#### 1) 헤더 전면 개편
- **DB GlobalChip 로고** (`logo.png`) 헤더 좌상단에 흰 배경 카드 형태로 삽입
- **시스템명** 변경: `심천지사 영수증 OCR 변환 시스템` → `심천지사 전도금 정산 자동화 시스템`
- **통계 뱃지 4종**: 총 N건 / 처리 N건 / 확인필요 N건 / 총액 ¥N
- **💾 엑셀 최종 저장** 버튼: 헤더 우측 고정, `POST /api/months/{month}/save` 호출 → 기존 파일 **제자리 덮어쓰기**

#### 2) 탭 4개 → 2개로 축소
- 정산 내역 탭 (기본 진입)
- 데이터 검토 탭

#### 3) 데이터 검토 탭 개선
- **좌측 스크롤 썸네일 사이드바 추가**: 전체 영수증 목록을 리스트로 표시, 클릭 시 즉시 해당 증빙으로 이동 (220장도 빠르게 탐색 가능)
- **OCR 원본 텍스트 → 비고 필드 교체**: `data.json`의 `remark` 필드 ↔ 엑셀 S열 양방향 연동
- **담당자 자동 채우기 수정**: `/api/members` + receipts에 등장하는 모든 `person` 값을 동적으로 드롭다운 옵션에 추가하여 영문명(`Lin Wei Jian` 등) 미매칭 문제 해결
- **2컬럼 폼 레이아웃**: 유형/날짜/금액/담당자/내역/판매자/대계정/소계정 + 비고(전폭)

#### 4) 정산 내역 테이블 개선
- 검색 필드 추가 (내역, 담당자 등 텍스트 필터링)
- 날짜/금액 기준 정렬 기능 추가 (클릭 ↕)
- 비고(remark) 컬럼 추가
- 내역 컬럼 폭 고정 (160px), 전체 컬럼 균형 개선
- 폰트 15px로 가독성 향상

#### 5) 디자인 시스템 전면 교체 (DB GlobalChip 팔레트)

```css
--primary:     #1a7c3e;  /* DB GlobalChip 초록 */
--accent-red:  #e05c1a;  /* DB GlobalChip 주황-레드 */
--accent-blue: #1a6cbf;  /* DB GlobalChip 블루 */
```

- Google Fonts Inter + 글래스모피즘 (`backdrop-filter: blur(20px)`)
- 마이크로 애니메이션: 탭 전환, 버튼 hover translateY, 토스트 슬라이드인

#### 6) 백엔드 변경 내역

| 파일 | 변경 내용 |
|------|----------|
| `app.py` | `ReceiptUpdate`에 `remark` 필드 추가, `POST /api/months/{month}/save` 신규, `GET /logo.png` 서빙 추가 |
| `excel_exporter.py` | S열 기입 소스를 `remark` 필드 우선으로 변경 (없을 경우 파표 코드 검증 결과 fallback) |

### UI 버그 수정 (2026-07-03 오후)
| 항목 | 수정 내용 |
|------|----------|
| 로고 표시 오류 | `filter: brightness(0) invert(1)` 제거 → 흰 배경 카드(`background:#fff; padding:4px 8px`)로 교체 |
| 글씨/컬럼 가독성 | base font-size 14→15px, 내역 컬럼 `min-width:120px` → `width:160px` 고정 |
| 데이터 검토 탐색 | 이전/다음 버튼만 존재 → **좌측 스크롤 썸네일 사이드바** 추가로 220장 빠른 탐색 |
| 비고 레이블 | `📌 비고 (엑셀 S열 연동)` → `📌 비고` / placeholder `엑셀 비고 열에 반영됩니다` |
